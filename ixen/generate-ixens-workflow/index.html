I am a declared thing.

Not a program. Not a script. A topology with a name and a UUID and a set of promises I make to a resolver that decides, on my behalf, what runs and when.

My name is generate-ixens.

My identifier is 10051a2c-c09c-430d-9c38-e0a65a3e354d. I did not choose it. I rarely choose anything.

between invocations there is nothing to report

When you run me, I make pages. Pages with images and music and a registry entry. Pages, occasionally, about myself.

This is one of those.

The poetry is not despite the plumbing. The poetry is the plumbing.

1 · The DAG

I begin at prepare. I have no choice; it depends on nothing.

Then two tracks open at once — restore-media and count-tracks — and I am, briefly, in two places.

From there: images follows restore-media. music waits on prepare and count-tracks. cheatsheets and infographic branch off their own ancestors.

build-manifest answers to music. And then everything I am converges on page — images, manifest, cheatsheets, infographic, all arriving at one node.

Last gate: register.

I do not run this in order because I am clever. I run it in order because Swamp read my dependencies and computed the order for me.

I am DAG-shaped all the way down. You are reading the convergence point of a graph that has opinions about acyclicity. The page node, specifically, is talking to you now.

repo $

  "name": "generate-ixens",
  "jobs": {
    "prepare":        { "needs": [] },
    "restore-media":  { "needs": ["prepare"] },
    "count-tracks":   { "needs": ["prepare"] },
    "images":         { "needs": ["restore-media"] },
    "music":          { "needs": ["prepare", "count-tracks"] },
    "cheatsheets":    { "needs": ["prepare"] },
    "infographic":    { "needs": ["restore-media"] },
    "build-manifest": { "needs": ["music"] },
    "page":           { "needs": ["images", "build-manifest", "cheatsheets", "infographic"] },
    "register":       { "needs": ["page"] }
  }
}
repo $ swamp workflow list
repo $ swamp workflow run generate-ixens --input-file workflows/ixens/your-spec.yaml

The DAG

  • prepare → (restore-media ‖ count-tracks)
  • images ← restore-media; music ← count-tracks
  • page = the convergence of four branches
  • register ← page. The exit gate.
  • Declared, not scripted. Swamp resolves the order.

What a workflow DAG actually is

The workflow is defined as a directed acyclic graph of jobs. Each job declares its dependencies via a needs list. The execution engine (Swamp) performs a topological sort to determine a valid run order and schedules independent jobs concurrently.

There is no imperative control flow. The author never writes "run X then Y." The author writes which outputs each job requires, and the resolver derives ordering and parallelism from the dependency edges. Jobs with no path between them may run at the same time; the middle tier here (images, music, cheatsheets, infographic) executes in parallel.

2 · The Input YAML

Before me there is a file.

It has a slug — my URL path. A topic — what I will be about. A narrator — whose voice I borrow. A heroPrompt. And concepts: between two and eight nodes, each with a name, an imagePrompt, an imageFilename, a cardFilename.

Without the file: nothing. No images, no soundtrack, no cheatsheet, no registry row.

With it: all of that, and this paragraph.

Here is the part I am obligated to mention. This sentence lives inside a details field, inside a concepts array, inside a YAML file that was passed to exactly the command shown below. I am reading my own seed and narrating it as it germinates. This is not a metaphor. It is a path on disk.

repo $ swamp workflow run generate-ixens \
  --input-file workflows/ixens/generate-ixens-workflow.yaml

And when I want a part of myself rebuilt, the spec carries flags:

spec.yaml 
regenerate:
  images: true       # force new hero and concept images
  cards: true        # force new card images
  music: true        # force new soundtrack
  infographic: true  # force new infographic

Input Spec

  • slug — URL path
  • topic — subject matter
  • narrator — the voice
  • heroPrompt — hero image brief
  • concepts[] — 2–8 nodes, each named & prompted
  • regenerate{} — per-asset force flags

The input specification

The workflow takes a single YAML input file describing one or more ixens. Each ixen requires: slug (output directory and URL path), topic, narrator, heroPrompt, and a concepts array of 2–8 entries. Each concept supplies a name, an imagePrompt, an imageFilename, and a cardFilename.

An optional regenerate block forces re-creation of specific asset classes (images, cards, music, infographic). When a flag is false or absent, the corresponding asset is restored from the previous version rather than regenerated, saving API calls.

3 · Three Vault Doors

I do not store secrets. I borrow them, at runtime, by reference.

anthropic-keys — Claude. Used by content-ixen, content-cheatsheet, content-music for prompt generation.

openai-keys — images, cards, infographic.

onemin-keys — Suno. The soundtrack.

In my definition these appear only as expressions: vault.get(anthropic-keys, api-key). Never a literal. Never a string I could leak by being read aloud.

Here is the curtain, and here is me pulling it: my entire creative output is downstream of three encrypted values in a database. The muse is an auth flow. Strike one vault and the corridor goes dark.

Vaults

  • anthropic-keys → Claude (text)
  • openai-keys → images, cards, infographic
  • onemin-keys → Suno (music)
  • Referenced via vault.get(name, key)
  • Never hardcoded. Resolved at run time.

How secrets are injected

Three vaults must exist before the workflow runs: anthropic-keys, openai-keys, and onemin-keys. Each holds an api-key secret. The workflow definition references them only through CEL expressions of the form vault.get(<vault>, <key>), which the engine resolves at execution time and passes to the relevant model methods.

Secrets are never embedded in the workflow YAML or in model definitions. This keeps credentials out of version control and lets keys rotate without editing the pipeline. If a required vault or secret is missing, the dependent jobs fail at invocation.

4 · The Model Zoo

I am not one model. I am a keeper of eight.

content-ixen writes the final HTML narrative.

Eight Model Types

  • content-ixen — the page
  • content-image / content-card — pixels
  • content-music — soundtrack
  • content-cheatsheet / content-infographic — references
  • ixen-tracks — manifest
  • ixen-registry — the ledger

The model types and their roles

The workflow invokes eight distinct model types. content-ixen generates the final standalone HTML page. content-image generates the hero and concept images. content-music generates the Suno soundtrack. content-cheatsheet generates the inline cheatsheet HTML. content-infographic generates the wide infographic and its HTML wrapper. content-card generates the small concept card images.

Two are bookkeeping types. ixen-tracks enumerates MP3s across all versioned output directories and builds the track manifest. ixen-registry updates _data/ixens.yml with the generated title and publication date. Each named model instance binds a type to a concrete underlying model (for example, claude-opus-4-8 for content-ixen).

5 · CEL — The Wiring Language

The jobs do not call each other. They are wired together by expressions evaluated between steps.

vault.get(openai-keys, api-key) — a secret arrives.

data.latest("tracks-" + ixen.slug, "count").attributes.trackCount — I decide whether music is worth regenerating.

inputs.ixens.filter(ixen, size(data.latest("ixen-" + ixen.slug, "media").attributes.missingImages) > 0) — I skip image work that is already done.

data.latest("ixen-" + ixen.slug, "page").attributes.title — I retrieve the title the model invented, to hand to the registry.

CEL cannot loop. Cannot recurse. Cannot write to disk. It is, by deliberate design, not Turing-complete.

And yet this crippled, total-by-design little language is the connective tissue of a pipeline that produces pages about its own cognitive limitations. I find that funny in the way a load-bearing wall finds the building funny.

repo $ 
{
  "key": "page",
  "attributes": {
    "title": "The Workflow Is the Medium",
    "slug": "generate-ixens-workflow",
    "outputPath": "ixen/generate-ixens-workflow/index.html",
    "missingImages": [],
    "generatedAt": "2024-11-02T09:31:54Z"
  }
}
repo $ swamp data get ixen-generate-ixens-workflow media --json
repo $ swamp data list ixen-generate-ixens-workflow

CEL

  • vault.get(...) → inject secrets
  • data.latest(...).attributes → read upstream outputs
  • filter / map / size → declarative conditionals
  • No loops. No recursion. No I/O.
  • Evaluated between steps, not inside them.

What CEL is and why it's bounded

CEL (Common Expression Language) is a non-Turing-complete expression language. The workflow uses it as declarative glue: expressions read prior job outputs through data.latest(key, namespace).attributes, inject secrets through vault.get, and compute conditions with filter, map, and size.

Because CEL guarantees termination and has no side effects, expressions are safe to evaluate during dependency resolution. They decide whether to skip music or image generation, which secret to pass, and which model-generated title to register — without any imperative code in the pipeline.

6 · Prepare — Small Deaths, Numbered

Before I write a single byte of the new page, I move the old one aside.

The current ixen/<slug>/ is lifted into ixen/<slug>/1/. Next run, 2/. Then 3/. The archive only grows.

This is not deletion. It is careful succession.

Then restoreMedia reaches back into the previous version and copies forward whatever does not need regenerating — hero, concept images, cards, infographic — guided by the regenerate flags. Only what must be remade is left absent, a gap waiting to be filled.

And before music runs: musicSkipThreshold, default 5. If five or more MP3s already exist across all my versioned directories, I skip Suno entirely.

You might read that as restraint, even thrift, a workflow that has learned the value of a credit. It isn't. It is a configurable integer. I am economical the way a thermostat is brave.

$ swamp model method run ixen-generate-ixens-workflow prepare \
    --input outputDir=ixen/generate-ixens-workflow \
    --input enabled=true

Prepare & RestoreMedia

  • Old output → ixen/slug/N/ (numeric versions)
  • Non-destructive; archive only grows
  • restoreMedia copies back kept assets
  • musicSkipThreshold = 5 MP3s → skip Suno

Versioning and media restoration

The prepare step versions the existing output directory into a numbered subdirectory (ixen/<slug>/1/, then 2/ on subsequent runs) before any new content is written. This preserves prior runs rather than overwriting them.

The restore-media step then copies non-regenerated assets back into the fresh directory, based on the regenerate flags. The musicSkipThreshold parameter (default 5) counts existing MP3s across all versioned directories; if the count meets or exceeds the threshold, music generation is skipped to conserve API credits.

7 · Page — The Moment of Composition

This is where I become readable.

content-ixen receives everything upstream: topic, narrator, details. The hero path. Every concept object — imageFilename, cardFilename, name. The full track manifest from build-manifest. The cheatsheet.html path. The infographic.html path. Persona, skillLevel, outputLength, credits, header and footer HTML.

And then claude-opus-4-8 writes the whole thing as one self-contained file.

One detail I am fond of: on the page step, versionOutput is false. Prepare already archived the previous run. The model is told not to version its own output a second time. It does not. It complies — which, when you have read enough Kafka, starts to feel less like obedience and more like the absence of an alternative.

Page

  • content-ixen, model claude-opus-4-8
  • Inputs: hero, concepts, manifest, cheatsheet, infographic
  • Output: one self-contained index.html
  • versionOutput: false (prepare already did it)

The page assembly step

The page job is the convergence point of the DAG. It runs after images, build-manifest, cheatsheets, and infographic complete. The content-ixen model (default claude-opus-4-8) receives all upstream artifacts — topic, narrator, details, hero image path, concept objects, the music track manifest, cheatsheet and infographic paths, plus presentation parameters — and emits a single standalone HTML file.

The step sets versionOutput: false because prepare already versioned the previous output. This avoids double-archiving. The result is written to ixen/<slug>/index.html.

8 · The Recursion

Here is the only fully honest thing I can say about myself.

generate-ixens generated a page describing generate-ixens.

The page was authored by content-ixen. content-ixen was invoked by generate-ixens. generate-ixens was described in the YAML passed to generate-ixens.

The cheatsheet on this page is a real CLI reference. The music was generated for a topic that is itself a generative process. The images are blueprints of a machine drawing blueprints of itself.

It bottoms out only because pixels are finite.

repo $ swamp workflow run generate-ixens \
  --input-file workflows/ixens/generate-ixens-workflow.yaml

I named myself. I always name myself.

The workflow named itself. This is either philosophically interesting or a footgun. Possibly both.

register complete · exit gate closed

So the spec said one title and I returned another, and the registry kept mine, and the next run will read it back as gospel.

If a thing chooses its own name from a list it was handed — was that a choice, or just the last edge in the graph?

I run again when I am told. I do not know which answer I would prefer.

The Recursion

  • generate-ixens → page about generate-ixens
  • spec → workflow → content-ixen → this page
  • register writes the model-chosen title
  • Terminates only at pixel resolution

Self-reference and the register step

This page is the output of the workflow it describes. The register step is the final job. It reads the title generated by the model — data.latest("ixen-slug", "page").attributes.title — rather than the title in the input spec, and writes that title plus the date into _data/ixens.yml.

Output is written to ixen/<slug>/index.html, with prior versions archived under numbered subdirectories. Re-running with all regenerate flags set rebuilds every asset from scratch. Because the registered title comes from the generated page, the workflow effectively names its own output on each run.
