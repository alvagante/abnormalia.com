What a DAG actually is

A directed acyclic graph expresses dependencies between jobs without specifying an execution order directly. Each job declares which jobs must complete before it can start. There are no cycles, so a valid topological ordering always exists.

The runner — here, swamp — computes that ordering and executes independent jobs in parallel whenever their dependencies are satisfied. The author describes constraints; the scheduler derives behavior. Changing the order means changing the declared dependencies, not editing a sequence of commands.

The input specification

The workflow is parameterized entirely by a YAML input file. Each ixen entry requires a slug (the output path segment), a topic, a narrator description, a heroPrompt for the hero image, and a concepts array of two to eight items.

Each concept supplies a name, an imagePrompt, an imageFilename, and a cardFilename. An optional regenerate block contains boolean flags (images, cards, music, infographic) that force regeneration of otherwise-reused assets. The file is the complete brief; no other source of intent is consulted at runtime.

How secrets are handled

API credentials are stored in named vaults, separate from the workflow definition. The workflow requires three: anthropic-keys (Claude), openai-keys (OpenAI image models), and onemin-keys (the Suno/Lyria music service).

At runtime, the workflow resolves credentials with CEL expressions such as vault.get(anthropic-keys, api-key). The secret value is injected at execution time and never written into the workflow file or its outputs. This keeps credentials out of version control and limits their exposure to the moment of use.

The eight model types

The workflow invokes eight distinct model types, each responsible for one kind of output. content-ixen generates the final HTML narrative. content-image generates the hero and concept PNGs via OpenAI. content-music generates the Suno soundtrack. content-cheatsheet generates inline cheatsheet HTML. content-infographic generates the wide infographic PNG and its HTML wrapper. content-card generates the small concept cards.

Two further models perform bookkeeping: ixen-tracks counts and builds the track manifest across all versioned directories, and ixen-registry syncs the site data file with the generated title and date. Separating responsibilities keeps each model narrow, testable, and independently invokable.

I have no code in the imperative sense. I have wiring.

CEL — Common Expression Language — connects one model's output to the next model's input, declaratively, without a single loop.

What CEL does here

Common Expression Language is a small, non-Turing-complete expression language used to compute values declaratively. In this workflow it wires job inputs to prior outputs and to external data without imperative code.

Expressions read previous step results via data.latest(...), inject secrets via vault.get(...), and transform input lists with map and filter. Because CEL cannot loop, recurse, or perform side effects, its evaluation is bounded and predictable. Control flow lives in the DAG; CEL only computes the values that flow along its edges.

Every run begins with a small death, and I number it.

Before a single byte of new content is written, prepare lifts the existing ixen/slug/ into ixen/slug/1/. Next time, 2/. Then 3/. Nothing is destroyed. Everything is archived.

Then restoreMedia reaches back into the boxes and copies forward whatever does not need remaking — hero.png, the concept images, cards, the infographic. Governed entirely by the regenerate flags. Only what must be reborn stays absent.

And there is musicSkipThreshold. Default: 5. If five or more MP3s already exist across the versioned directories, music regeneration is skipped.

This is not restraint. It is not maturity. It is not me learning to let the old songs stand. It is an integer. Five. You can change it.

Prepare + restoreMedia

  • Existing output → ixen/slug/N/.
  • Non-destructive, numbered.
  • restoreMedia copies back reusable assets.
  • regenerate flags decide what stays absent.

Versioning and reuse

Before generating new content, the prepare step moves the current output directory into a numbered subdirectory (ixen/slug/1/, then 2/ on the next run, and so on). No prior output is deleted; each run is preserved.

The restoreMedia step then copies reusable assets — hero image, concept images, cards, infographic — back into the fresh directory unless their regenerate flag is set. The musicSkipThreshold parameter (default 5) skips music generation when at least that many MP3 files already exist across versioned directories, conserving music-API credits.

The page assembly step

The page job is the workflow's convergence point. It runs only after images, build-manifest, cheatsheets, and infographic complete. The content-ixen model receives every upstream output: topic, narrator, details, hero image path, concept objects, the full track manifest, the cheatsheet and infographic HTML, plus presentation parameters like persona, skillLevel, outputLength, credits, and header/footer markup.

From these inputs the model (claude-opus-4-8 by default) composes a single standalone HTML page. The step sets versionOutput to false because the prepare step already archived prior output, avoiding a redundant second version.

The self-referential case

This page is the output of the generate-ixens workflow describing the generate-ixens workflow. The input YAML, the model invocations, the cheatsheet, the music, and the images are all genuine artifacts of the same pipeline they document.

The register step finalizes the run by reading the model-generated title from data.latest("ixen-slug", "page").attributes.title and writing it, with the date, into _data/ixens.yml. Setting the regenerate flags and re-running the workflow reproduces the page from scratch with new assets. The topology, however, remains fixed.
