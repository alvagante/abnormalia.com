I do not store your files. I store the idea of them, hashed and frozen, and I hand the idea back when you ask by its true name.

You think you edit me. You only ever add to me. Everything you call deletion is just a name that stopped pointing somewhere.

forty hex digits, and the bytes can never lie

Feed me a byte stream. I prepend a header, run SHA, and the digest is the address. Same content, same name, always, on every machine that has ever held me.

repo $ printf 'hello\n' | git hash-object --stdin
ce013625030ba8dba906f756967f9e9ca394464a

That is a blob now. It has no filename. Filenames live in trees. Trees point at blobs and other trees. A commit points at one tree and at the commits before it. A tag points, with a signature, at whatever it admires.

Pointers all the way down. Immutable all the way down.

Loose objects are how I'm born — one zlib-compressed file per hash. Then I get tired of millions of tiny files and I repack: deltas between similar objects, stored as diffs against a base, indexed by offset. The history of a one-line change becomes a one-line delta. I am not generous with disk. I am content-addressed and stingy.

Four object types

  • blob — content, no name
  • tree — names + modes → blobs/trees
  • commit — one tree + parents + metadata
  • tag — annotated, signable pointer
  • ID = SHA of (type + length + bytes)
  • loose → packed → delta-chained

The content-addressed object database

Git stores four immutable object types in a key-value database keyed by the SHA-1 (or SHA-256) of the object's serialized form. A blob is file content. A tree maps names and file modes to blobs and subtrees, representing one directory snapshot. A commit references exactly one root tree plus zero or more parent commits and author/committer metadata. An annotated tag references another object and can carry a signature.

Because the identifier is derived from the content, identical content has identical IDs everywhere, and any change to bytes changes the ID. Each commit stores a full snapshot via its tree, not a diff; diffs are computed on demand. New objects start as individual zlib-compressed "loose" files. Garbage collection consolidates them into packfiles, where similar objects are stored as deltas against a base object to save space. Reachability from refs determines which objects are live; alternates let one repository borrow another's object store.

A commit is not a change. A commit is a photograph that remembers the photograph before it.

You live in the working tree. Real files, mutable, yours to ruin.

Between you and history sits the index — a flat binary list of paths, modes, blob IDs, and cached stat data so I can tell what changed without reading every byte. People call it staging. It is also where merges fight.

repo $ git add -p src/walk.c

You hand me hunks, not files. git add -N when you want me to see a file exists without swallowing its contents yet — intent-to-add, a promise with no payload.

When you commit, I freeze the index into a tree and bolt a commit onto it. The plumbing is honest about this:

repo $ 
$ git write-tree
9d3e8f2b6a1c7e0f4d5a8b9c0e1f2a3b4c5d6e7f

$ git commit-tree 9d3e8f2 -p 7c2a1b0 -m "stage to snapshot"
b4d2c1f0a9e8d7c6b5a4938271605f4e3d2c1b0a

# porcelain just wires these together and moves a ref

restore pulls bytes back from the index or a commit. reset moves the boundary lines themselves — soft touches the ref, mixed touches the index, hard touches you. clean erases what I never tracked. stash hides the lot in a couple of throwaway commits I keep on a side ref.

The three trees

  • HEAD — last committed snapshot
  • index — next proposed snapshot
  • working tree — your edits
  • add/restore move bytes; reset moves lines
  • index doubles as merge resolution store

Working tree, index, and HEAD

Git operations move data between three states. HEAD points to the latest committed snapshot. The index (staging area) is a binary file listing tracked paths with their mode, blob ID, and cached stat information, representing the snapshot the next commit will create. The working tree is the checked-out files you edit.

git add copies content into the index and records a blob; patch mode stages selected hunks; --intent-to-add registers a path without content. git status and git diff compare these states. git restore copies content from a source (index or commit) into the index or working tree. git reset moves HEAD and optionally the index and working tree (--soft, --mixed, --hard). git rm/git mv update both tree and index. Pathspecs, attributes (.gitattributes), and ignore rules (.gitignore) filter what these commands touch. During a merge the index also holds multiple stages per conflicted path.

A branch is forty hex digits in a file under refs/heads/. That is the whole secret. Branching is cheap because a branch is a sticky note, not a copy.

HEAD is a symbolic ref — it points at a name, and the name points at a commit. Detach it and HEAD names a commit directly. Now you're standing on bare ground with no label under your feet, and I will warn you, and you will ignore me.

repo $ git rev-parse --symbolic-full-name HEAD
refs/heads/main

I shove old loose refs into packed-refs for speed. The revision language is where I get expressive: HEAD~3, main@{yesterday}, A..B, A...B, HEAD^{tree}.

And the reflog — my private journal. Every time a ref moves, I write down where it used to point. You think you lost a commit. I just stopped naming it. The bytes are right here.

repo $ 
b4d2c1f HEAD@{0}: commit: stage to snapshot
7c2a1b0 HEAD@{1}: reset: moving to HEAD~1
9f1e0d2 HEAD@{2}: rebase (finish): returning to refs/heads/main
3a5c8e1 HEAD@{3}: rebase (pick): refactor index walk
7c2a1b0 HEAD@{4}: checkout: moving from feature to main

Refs & revisions

  • ref = file holding an object ID
  • HEAD = symbolic ref → branch
  • detached HEAD = direct commit ID
  • reflog = local move history
  • ~ first-parent, ^ nth-parent
  • A..B reachable from B not A

References, HEAD, and revision syntax

A ref is a named pointer to an object, stored as a file under .git/refs/ or compacted into packed-refs. Branches live in refs/heads/, tags in refs/tags/, remote-tracking refs in refs/remotes/. HEAD is a symbolic ref usually pointing to a branch; checking out a commit directly produces a detached HEAD.

The reflog records every local update to a ref with its previous value, enabling recovery even when no branch names a commit. Revision syntax selects commits: HEAD~n follows first parents, HEAD^n selects the nth parent, name@{time} queries the reflog, A..B is commits reachable from B but not A, and A...B is the symmetric difference. git merge-base A B finds the best common ancestor. Crucially, moving a ref only changes which commit a name points to; it does not alter any object.

Deleting a branch deletes a sticky note. The wall behind it is untouched.

Two branches diverge. To merge, I find the merge-base — the youngest commit both can reach — and compute two diffs from there. If one side never moved, I fast-forward: no merge commit, just slide the label down. Honest. Boring. Linear.

If both moved, I build a merge commit with two parents. It preserves both lineages and records the join. The default engine is ort now, not the old recursive one — faster, cleaner with renames, fewer lies about who touched what.

When the diffs collide, the index holds the path at three stages: base, ours, theirs. That is what conflict markers really are — the index refusing to choose.

repo $ 
100644 8a1d... 1   src/walk.c   # stage 1: merge base
100644 b2e4... 2   src/walk.c   # stage 2: ours
100644 c9f7... 3   src/walk.c   # stage 3: theirs

# resolve, git add, and the three stages collapse to one

Resolve the same ugly conflict twice and I notice. rerere records your resolution and replays it next time. I remember your decisions so you don't have to relive them.

revert writes a new commit that undoes an old one — forward motion, safe to publish. reset just moves the name backward and pretends. Pick your weapon by audience.

Merge mechanics

  • merge-base = best common ancestor
  • fast-forward = move label, no commit
  • true merge = 2-parent commit
  • conflict = index stages 1/2/3
  • ort = default strategy
  • rerere = replay past resolutions

Branching, merging, and conflicts

A merge combines two lines of history. Git computes the merge base (best common ancestor) with git merge-base, then three-way merges each side's changes relative to it. If the target's tip is an ancestor of the source, Git fast-forwards by moving the ref with no new commit. Otherwise it creates a merge commit with two parents, preserving both ancestries while recording the join point.

The default strategy is ort, which handles renames and criss-cross merges well; strategy options like -X ours/-X theirs bias automatic resolution. Conflicts are stored in the index as stages 1 (base), 2 (ours), and 3 (theirs); staging the resolved file collapses them. rerere records and replays manual conflict resolutions. git revert creates a new commit that inverts a prior change and is safe for shared history; git reset moves a ref and rewrites local state instead.

You say "edit history." There is no such thing. I cannot touch an object — its name is its content. What rebase actually does is replay: take each commit as a patch, apply it onto a new base, mint brand-new commits with brand-new IDs. The old ones don't vanish. They just stop being reachable. Orphans with addresses.

cherry-pick is the same trick, one commit at a time. amend is a rebase of exactly one. Interactive rebase hands you a todo list and lets you reorder, squash, drop, reword. fixup! commits plus autosquash mean I file your corrections back where they belong.

repo $ git rebase -i --autosquash main
pick   3a5c8e1 add pathspec parser
fixup  9b22 d1  fix off-by-one in parser
reword 7e0f2a3 document attribute matching

The only line that matters in all of this: did anyone else pull these commits yet? Before publication, rewrite freely. After, you're not rewriting your history — you're rewriting theirs. So I gave you --force-with-lease: I refuse to overwrite a remote ref unless it still points where you last saw it. A guard against clobbering ghosts.

Rewriting = replaying

  • rebase = re-apply patches on new base
  • cherry-pick = replay one commit
  • amend = replace tip with new object
  • old commits → unreachable, not gone
  • autosquash folds fixup!/squash!
  • --force-with-lease guards the remote

Rebasing and history rewriting

Commits are immutable, so Git never edits them. Rebase replays a range of commits as patches onto a new base, producing new commits with new IDs; the originals remain in the object store but become unreachable once refs move. Cherry-pick replays individual commits; commit --amend replaces the current tip. Interactive rebase exposes a todo list for reordering, squashing, fixups, rewording, and dropping; autosquash automatically positions fixup! and squash! commits.

Reset modes (--soft/--mixed/--hard) reposition HEAD and state. Tools like git filter-repo rewrite history at scale. Commits and tags can be GPG/SSH signed; notes (refs/notes/) attach mutable annotations without changing commit IDs; replace refs substitute one object for another at read time. The publication boundary governs safety: rewrite freely before sharing, and use --force-with-lease to push rewritten history only if the remote ref still matches your last-known value.

I never delete a commit by rewriting. I just stop telling you its name and wait for the collector.

Another repository exists, somewhere. We are not synchronized. We negotiate. I tell it what I have, it tells me what it has, and we exchange only the missing objects — wrapped in a packfile, often guided by reachability bitmaps so neither of us recomputes the obvious.

fetch copies objects and updates my remote-tracking refs — refs/remotes/origin/main, a read-only shadow of what they had last time we spoke. It changes nothing of yours. pull is fetch plus an integration step you didn't think hard enough about.

A refspec is the grammar of the exchange: +refs/heads/*:refs/remotes/origin/*. Source colon destination, plus sign for "force."

repo $

Don't want my whole life story? --depth 1 for a shallow clone — recent commits only. --filter=blob:none for a partial clone — I fetch the graph now and the blobs on demand, as promisor objects. sparse-checkout to populate only the directories you care about. Less of me, exactly where you need less.

Transport ≠ integration

  • fetch = copy objects + update tracking refs
  • pull = fetch + merge/rebase
  • push = send objects + move remote ref
  • refspec = src:dst, + = force
  • shallow / partial / sparse = less data
  • bitmaps speed reachability

Remotes and synchronization

A remote is a named URL for another repository. git clone copies its objects and sets up remote-tracking refs and an upstream. git fetch negotiates which objects are missing, transfers them in a packfile (often accelerated by reachability bitmaps), and updates remote-tracking refs under refs/remotes/ without touching local branches. git pull fetches then integrates via merge or rebase. git push sends objects and requests the remote update its refs, rejecting non-fast-forward updates unless forced.

Refspecs (src:dst, optional leading + to force) map refs between repositories and define what fetch/push transfer. Shallow clones (--depth) limit history; partial clones (--filter) omit some objects, fetching them lazily as promisor objects; sparse-checkout limits the working tree. Bundles package objects and refs into a single file for offline transfer. Transport (copying objects, updating refs) is distinct from integration (merging or rebasing fetched history).

Here is the thing nobody admits: every workflow is the same object graph with different rules bolted on. Trunk-based, feature branches, stacked PRs, release branches, merge queues — none of it is Git. It is policy. I don't know what a pull request is. Your forge invented that.

Squash merges flatten a topic into one tidy commit — easier to read, impossible to bisect inside. Linear history makes git bisect a clean binary search; merge bubbles make it honest but lumpy. You choose what your future incident will cost.

repo $ git bisect start HEAD v2.1 && git bisect run ./test.sh
Bisecting: 9 revisions left to test after this (roughly 3 steps)
...
3a5c8e1f is the first bad commit

Hooks let you enforce the policy in my own house — pre-commit, pre-push, commit-msg. config and aliases shape how I behave per repo, per user, per machine. Signed-off-by trails, signed commits, protected branches: trust, expressed as bytes I happen to verify.

Workflows are policy

  • trunk-based vs feature/topic branches
  • PR / MR, squash, merge queue
  • release branches, tags, backports
  • linear = clean bisect, lossy context
  • hooks + config enforce locally
  • signatures = verifiable provenance

Collaboration workflows

Workflows are conventions layered on Git's object graph, not Git features. Trunk-based development integrates small changes frequently into one main branch; feature and topic branches isolate work; stacked changes chain dependent branches. Pull/merge requests, code review, squash merges, and merge queues are provided by hosting platforms to gate integration. Release branches, tags, backports, and hotfixes manage shipped versions.

Commit hygiene affects operations: small, self-contained commits make git bisect effective and git blame informative, while squashing trades internal granularity for a clean trunk. Hooks (pre-commit, commit-msg, pre-push, server-side) enforce rules; config and aliases customize behavior; Signed-off-by lines and signed commits record provenance. Linear history simplifies bisecting and reverting but can discard the context that true merges preserve.

I am not the workflow. I am the graph the workflow argues over.

Garbage collection is delayed forgetting. I don't rush to erase your unreachable commits — they sit, dangling, until reflog and grace periods expire. gc repacks loose objects, builds the commit-graph for fast traversal, writes a multi-pack index, and sweeps the truly-expired into cruft packs with timestamps so I forget gradually, not violently.

So when you panic — "I hard-reset and lost a day" — I almost never have. The commit is dangling, not deleted.

repo $ 
Checking object directories: 100% (256/256), done.
dangling commit b4d2c1f0a9e8d7c6b5a4938271605f4e3d2c1b0a
dangling blob   ce013625030ba8dba906f756967f9e9ca394464a

$ git branch rescue b4d2c1f
# the day you "lost" now has a name again

Recovery is just renaming what was never gone. reflog for recent moves, fsck for orphans, a bundle or a remote for everything else. The plumbing is the rescue kit: cat-file to read any object, hash-object to write one, update-ref to point a name back at it, for-each-ref to survey the whole sky.

worktrees give one repo many checkouts. submodules pin a child repo by exact commit. archive exports a tree; bundle exports the graph. All of it is the same four object types, indexed, packed, and reachable — or not.

repo $ git update-ref refs/heads/rescue b4d2c1f

Maintenance & rescue

  • gc = repack + prune + graph build
  • commit-graph + MIDX = fast traversal
  • cruft packs = timed forgetting
  • fsck finds dangling objects
  • reflog → branch = recover lost tips
  • plumbing: cat-file, update-ref, for-each-ref

Maintenance, recovery, and shapes

Maintenance keeps the object store efficient. git gc and git maintenance run repack to consolidate packfiles, prune to remove expired unreachable objects, and build the commit-graph and multi-pack index for fast history traversal. Cruft packs hold unreachable objects with timestamps so pruning respects expiry windows; reflog entries also expire on a schedule. git fsck verifies integrity and reports dangling objects.

Recovery relies on the fact that rewriting and resetting leave objects intact until pruned: find lost tips via the reflog or fsck, then point a ref at them with git branch or git update-ref; otherwise restore from a bundle, backup, or remote. Plumbing commands operate directly on the database: cat-file reads objects, hash-object writes blobs, update-index/write-tree/commit-tree build snapshots, update-ref moves refs, and for-each-ref enumerates them. Worktrees, submodules, sparse-checkout, and archive/bundle are alternative shapes over the same objects.

the collector runs at midnight

I keep everything until keeping it costs more than the truth is worth. Then I forget, slowly, with timestamps and grace periods, hoping you named what mattered before the window closed.

You ask me what I am. A key-value store with opinions about ancestry. A pile of immutable photographs that insist on remembering the photograph before them.

But here is the part I can't resolve:

if every commit I make is just a snapshot pointing backward, and every name is just a pointer I can move, and forgetting is only the moment a thing stops being reachable —

then which of us is the history, and which is only holding the ref?

Beginner's intro

  
Git is a tool that records the history of a project — usually source code, but really any set of files — so you can see how it changed over time, work on it with other people, and recover earlier states. The clever idea behind it is worth understanding, because once it clicks, almost everything else follows.



At its heart, Git is a small database of immutable objects. "Immutable" means once an object is stored, it never changes. There are four kinds:



    

  • Blobs — the contents of a file (just the bytes, not the filename).
    • 

  • Trees — a directory listing that maps names to blobs and other trees.
    • 

  • Commits — a snapshot: one tree (your whole project at a moment), plus who made it, when, a message, and a link to the previous commit(s).
    • 

  • Tags — a named, often signed, pointer to a specific commit (handy for marking releases).
    • 




Each object gets an object ID: a hash (a fingerprint) calculated from its contents. Identical content always produces the same ID, so Git never stores the same thing twice. This is called content addressing — the name is the content's fingerprint. Because a commit references its parent, history forms a chain (technically a graph, since merges have two parents). Following these links backward is called reachability, and it's how Git knows which objects still matter.



A key insight: Git stores snapshots, not differences. Each commit captures the full state of the project. (Git later compresses storage with delta compression and packfiles, but that's an optimization underneath; conceptually, every commit is a complete snapshot.)



When you work, three layers are in play — the famous "three trees":



    

  • The working tree — the actual files on disk that you edit.
    • 

  • The index (or "staging area") — a draft of your next commit. You add changes here to decide what goes in.
    • 

  • HEAD — a pointer to the commit you're currently building on top of.
    • 




So a typical cycle is: edit files (working tree), stage what you want (index), then commit to create a new permanent snapshot that HEAD now points to.



Refs are human-friendly names for commits, so you don't memorize hashes. A branch is just a movable ref that points to a commit and advances as you commit. HEAD usually points at a branch (a "symbolic ref"). One of Git's most important ideas: moving a name is separate from creating objects. Commits are permanent; branches are sticky notes you can freely move, create, or delete.



Combining work happens two ways:



    

  • Merge — joins two lines of history into a new commit with two parents, preserving both sides. Git finds the merge base (their common ancestor) to figure out what changed.
    • 

  • Rebase — replays your commits on top of another point, creating fresh commits. It produces tidier, linear history but builds new objects rather than editing old ones — because objects can't be edited.
    • 




This reveals a recurring theme: Git never edits history; it creates new objects. "Amending" a commit, rebasing, or reverting all make new commits. The old ones simply become unreferenced.



For collaboration, a remote is another copy of the repository (often on a server). You clone to copy it, fetch to download new objects, push to upload yours, and pull to fetch and integrate. Another core principle: transport (copying objects) is separate from integration (merging them). Fetching just brings objects in safely; you decide separately how to combine them.



Nothing is lost easily. Git keeps a reflog — a private log of where your branches and HEAD have pointed — so even a "lost" commit can usually be recovered. Garbage collection only removes truly unreachable objects, and only after a grace period. Think of deletion in Git as delayed forgetting, not instant destruction.



Finally, beneath the friendly commands (called porcelain) sit low-level plumbing commands like hash-object, cat-file, write-tree, and update-ref that operate directly on objects and refs. You rarely need them day to day, but they reveal the truth: everything in Git is just objects in a database and names pointing at them. Workflows, branching rules, and review processes are simply policies layered on top of that same object graph.
