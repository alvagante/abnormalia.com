You boot the machine and you think you have reached the metal. You haven't. I got there first. I claimed the privileged ring, set the IOMMU, carved the firmware's memory map into something I could ration — and only then did I hand you a kernel and let you believe it was alone.

Who lives where

There is a line through everything I touch. Above it: domains, plural, each convinced of its own completeness. Below it: me, and the hardware I refuse to share.

dom0 is the firstborn. It holds the toolstack, the real drivers, the right to ask me for things no guest may ask. A domU asks for nothing dangerous — it only thinks it does. When it wants the disk, it is really asking dom0, and dom0 is really asking me.

I split work further when fear demands it. A driver domain owns a NIC so that a crashing network stack takes down a guest and not the world. A stub domain carries a guest's device emulation, so the QEMU that pretends to be its chipset never runs inside dom0 at all.

Because if dom0 falls, the management plane falls with it — and the one component I extended the most rope to is the one holding the knife.

Domain model

  • Hypervisor: scheduling, memory, isolation. Nothing else.
  • dom0: real drivers, toolstack, privileged hypercalls.
  • domU: unprivileged guest, mediated everything.
  • Driver domain: hardware ownership, blast-radius limiting.
  • Stub domain: per-guest device model, kept out of dom0.
  • Authority lives in capabilities, not code size.

The domain model and privilege boundaries

Xen is a type-1 hypervisor: a thin layer that runs at the highest hardware privilege and owns CPU scheduling, memory allocation, and inter-domain isolation. It deliberately contains no device drivers, no filesystem, and no general-purpose userspace.

dom0 is the initial privileged domain. It runs a normal OS kernel, holds the physical device drivers, and runs the toolstack (xl/libxl) used to create and control other domains. Through privileged hypercalls it can map other domains' memory and manage hardware.

domU domains are unprivileged guests. They cannot touch hardware directly; their I/O and control requests are mediated by dom0 or by delegated domains and ultimately by the hypervisor.

A driver domain is an unprivileged domain granted ownership of a specific physical device, so a driver failure is isolated from dom0. A stub domain hosts the per-guest device emulator (QEMU) outside dom0 to reduce the trusted computing base. Because dom0 concentrates privilege and drivers, it is simultaneously the control plane and the largest single point of compromise.

How my guests run

Some of my guests know they are guests. Some refuse to. The difference is how they reach me.

The old paravirtual ones speak to me directly — no pretending. They issue hypercalls where a bare-metal kernel would issue privileged instructions. Honest, fast, and a maintenance burden no one loved, because they need a kernel modified to know I exist.

The fully virtualized ones believe they own a whole computer. I let them. I build a fake chipset out of emulation and hardware assists, and when they try a privileged instruction the CPU traps down to me like a confession. Compatible with anything. Expensive at the seams.

And then PVH — the compromise I prefer. Hardware virtualization for the CPU and memory, paravirtual everything for I/O, no emulated firmware pretending to be a 1990s PC. Less surface to defend.

Every emulated device is a lie I have to keep consistent forever.

PV / HVM / PVH

  • PV: modified kernel, hypercalls, no HW assist needed.
  • HVM: unmodified guest, VT-x/AMD-V, emulated platform.
  • PVH: HW-virtualized CPU/MMU, PV I/O, minimal firmware.
  • Tradeoff axis: compatibility ↔ performance ↔ attack surface.
  • Emulation is the cost of pretending to be hardware.

Boot modes and guest execution

PV (paravirtualization) requires a guest kernel aware of Xen. It runs deprivileged and replaces privileged operations with hypercalls. It needs no CPU virtualization extensions but cannot run unmodified operating systems.

HVM (hardware virtual machine) uses Intel VT-x or AMD-V to run unmodified guests. The hypervisor and an emulator (QEMU, often in a stub domain) provide a virtual platform: BIOS/UEFI firmware, emulated chipset, and emulated devices. Privileged instructions trap to the hypervisor. Compatibility is high; emulation adds overhead and attack surface.

PVH combines hardware-assisted CPU and memory virtualization with paravirtual I/O and boot. It omits most legacy device emulation and heavyweight firmware, reducing the trusted code base while keeping good performance. Modern HVM guests typically also use PV drivers for disk and network, blurring the historical line between modes.

Time, divided

A guest sees four vCPUs and believes it has four. It has four promises. The physical cores are mine, and there are fewer of them than the sum of everything I've promised.

Scheduling

  • vCPUs are entitlements; pCPUs are the only reality.
  • Credit2: credit accrual + per-runqueue selection.
  • Pinning / affinity: hard limits vs. soft preference.
  • NUMA locality protects memory latency.
  • Steal time = vCPU runnable but not running.
  • Capacity planning ≠ guest-visible vCPU count.

CPU scheduling and vCPU topology

Xen schedules virtual CPUs (vCPUs) onto physical CPUs (pCPUs). The default Credit2 scheduler assigns each vCPU credit based on weight; runnable vCPUs are ordered within per-runqueue structures and selected by remaining credit, with preemption to maintain fairness and responsiveness. The older Credit scheduler used periodic credit allocation across all pCPUs.

Hard affinity restricts which pCPUs a vCPU may run on; soft affinity expresses a preference, important for keeping vCPUs near their NUMA-local memory to avoid cross-node access latency. Pinning trades flexibility for predictability.

Overcommit means total vCPUs exceed pCPUs, relying on guests being idle at different times. When a vCPU is runnable but cannot get a pCPU, the guest accrues steal time. Capacity planning must account for actual contention and latency targets, not just the vCPU counts advertised to guests.

Whose memory

A guest names a page. It says "frame 4096" with total confidence. That number means nothing to the hardware. It is a guest-physical address — a fiction I maintain — and I am the one who knows which machine frame it really lands on.

Between the guest's idea of memory and the silicon's, there is a translation that I never let the guest write directly. Either I shadow its page tables, or the CPU's nested paging does the second translation for me. Either way: the guest proposes, I dispose.

And memory moves. Through the balloon driver, a cooperating guest inflates — pins pages and hands them back — and I lend that reclaimed memory to a neighbor that needs it more. It is polite theft, agreed to in advance.

dom0 # xl mem-set web-frontend 1536
# web-frontend balloons down 512 MiB; pages return to the heap

When a domain dies I do not simply forget its pages. I scrub them — overwrite the dead memory before any other guest can read what it left behind. Privacy here is a loop that writes zeros.

Grants are not how I share memory. They are how I let one guest hand a single key to another, and watch them use it.

Memory

  • MFN (machine) vs. GFN (guest-physical) — I own the map.
  • Shadow page tables or HAP/EPT/NPT for translation.
  • Ballooning: cooperative reclaim and redistribution.
  • Scrubbing: zero freed pages before reuse.
  • Grants: scoped, revocable inter-domain page access.

Memory ownership and translation

Physical RAM is addressed by machine frame numbers (MFNs). Each guest sees guest physical frame numbers (GFNs) that the hypervisor maps onto MFNs. Guests are never allowed to install raw page tables that reference machine addresses directly.

Translation is mediated either by shadow page tables maintained by Xen, or by hardware-assisted paging (Intel EPT / AMD NPT), which performs guest-physical to machine translation in hardware. Hardware paging is the common modern path.

The balloon driver lets a guest voluntarily release pages back to Xen so memory can be redistributed, enabling overcommit. Under memory pressure this can degrade guests that cannot give pages back. Freed pages are scrubbed (zeroed) before reallocation to prevent data leakage. Grant tables are a separate, fine-grained mechanism by which one domain authorizes another to map or transfer specific pages, used mainly for I/O rather than for general memory sharing.

I/O is a conversation between strangers

No guest of mine touches a disk. It touches blkfront, a frontend that writes a request into a shared ring, grants the backend access to the buffer, and rings a bell.

The bell is an event channel — a virtual interrupt I deliver. On the far side, blkback in dom0 or a driver domain wakes, reads the ring, maps the granted pages, performs the real I/O, and signals back. Two kernels, one circular buffer, and me carrying the notification across.

They agree on where to meet through xenstore — a little shared tree of strings where frontend and backend negotiate state: 1 initialising, 4 connected. When something hangs, it is almost always here: a backend stuck, a state field frozen mid-handshake, a ring with requests no one will ever consume.

Split I/O

  • Frontend (blkfront/netfront) in the guest.
  • Backend (blkback/netback) in dom0 or driver domain.
  • Shared ring: producer/consumer request descriptors.
  • Grant refs: buffer access; event channels: notification.
  • xenstore: discovery + connection state machine.
  • Most stalls = backend hang or mismatched state.

I/O paths and split drivers

Xen uses a split-driver model. A lightweight frontend driver in the guest (e.g. blkfront, netfront) cooperates with a backend driver (blkback, netback) in dom0 or a dedicated driver domain that owns the real hardware.

The two halves communicate through a shared memory ring buffer of request/response descriptors. The data buffers themselves are shared via grant references, which authorize the backend to map specific guest pages. Event channels deliver asynchronous notifications (virtual interrupts) so each side knows when to process the ring.

xenstore is a hierarchical key/value store used for device discovery and connection negotiation. Frontend and backend advance through a documented state machine (e.g. state 1 initialising, 4 connected). Many operational failures present as stalled backends, unconsumed ring entries, or a connection state that never reaches "connected," often due to a hung backend or version/feature mismatch.

When a guest touches real hardware

Sometimes a guest needs the device itself — the actual NIC, the actual GPU — not my polite emulation. So I hand it over. And the moment I do, I have a problem I cannot solve alone.

A real device can do DMA. It writes to physical memory by machine address, and it does not ask me first. Software isolation ends where the bus begins. So I lean on the IOMMU: I program its tables so that device can only reach the guest's own pages, and I remap its interrupts so it cannot inject a vector meant for someone else.

SR-IOV lets one physical function split into many virtual functions — a NIC pretending to be a dozen NICs, each assignable, each behind its own translation. Beautiful, until a device resets badly on reassignment, or its firmware does something the datasheet never mentioned.

Past the IOMMU, isolation stops being my promise and starts being the hardware's. I can only hope the silicon meant it.

Passthrough

  • Assigned device does real DMA — bypasses software.
  • IOMMU (VT-d/AMD-Vi) confines DMA to guest pages.
  • Interrupt remapping prevents cross-domain injection.
  • SR-IOV: physical function → many virtual functions.
  • Reset/FLR and firmware quirks break clean handoff.
  • Isolation becomes a hardware trust assumption.

Device assignment and DMA isolation

PCI passthrough assigns a physical device directly to a guest for near-native performance. Because the device performs DMA using machine addresses, software cannot mediate each access; isolation depends on the IOMMU (Intel VT-d, AMD-Vi). Xen programs IOMMU page tables so the device can only reach the assigning guest's memory, and uses interrupt remapping to ensure the device cannot deliver interrupts to other domains.

SR-IOV capable devices expose multiple virtual functions from one physical function, allowing many guests to share hardware with separate DMA contexts. Correct operation depends on reliable device reset (e.g. Function Level Reset) so no residual DMA or state leaks across reassignment.

In practice, passthrough security and stability rest on hardware and firmware behaving as specified. Buggy reset, shared interrupt lines, or firmware quirks can undermine containment, making performance isolation ultimately a hardware trust problem.

Carrying a guest across

A guest keeps running while I copy it out from under itself. This is the trick I'm proudest of and most afraid of.

I send its memory to the other host while it still executes — first pass, every page. Then I track which pages it dirtied behind me and send those again. And again. Each round smaller, chasing a moving target.

If the dirty rate stays high — a guest furiously rewriting the same pages — I never converge by copying alone. So at last I stop the world: pause the vCPUs, ship the final dirty pages and the register state, and resume on the far side. A pause measured in milliseconds, if I'm lucky and the workload is kind.

dom0 # xl migrate db-primary host-b
migration target: host-b
Saving to migration stream ...
xc: precopy_policy: iter 1 dirty 2097152 ...
xc: precopy_policy: iter 6 dirty 4096, suspending
Domain resumed on host-b.

Storage I do not move — it was shared or replicated all along. The MAC follows the guest, a gratuitous ARP teaches the switches where it lives now, and the TCP connections never knew. But the clock stutters across the pause, and somewhere a guest's monotonic time learns that monotonic was always my courtesy.

Live migration

  • Pre-copy: send all pages, then iterate on dirtied ones.
  • Convergence depends on dirty rate vs. link bandwidth.
  • Brief stop-and-copy: final pages + vCPU/device state.
  • Storage shared/replicated; not transferred inline.
  • MAC + gratuitous ARP keep network identity.
  • Exposes guest/host/fabric coupling and clock skew.

Live migration and state transfer

Save/restore serializes a guest's memory and CPU/device state to a stream. Live migration does this while the guest runs, using pre-copy: an initial pass copies all memory to the destination, then successive passes copy only pages dirtied since the previous pass.

If the dirty page rate is high relative to available bandwidth, iterations stop shrinking. The hypervisor then briefly suspends the guest (stop-and-copy) to transfer the remaining dirty pages and the final vCPU and virtual device state, then resumes it on the destination. Downtime is typically milliseconds for well-behaved workloads.

Disk state is not copied inline; migration assumes shared or replicated storage. Network identity is preserved by keeping the MAC address and issuing a gratuitous ARP so switches relearn the location. Migration can expose hidden dependencies: clock discontinuities, latency to storage or the network fabric, and assumptions about host-local resources.

The failures have shapes I recognize. A guest that won't stop spinning, eating credit its neighbors needed — the noisy neighbor, and steal time everywhere downstream. An IRQ no driver claims, storming until I disable it and something quietly loses its interrupts. A backend that stalled, leaving a ring full of requests and a frontend timing out into the dark.

And the failure I do not survive: dom0 compromised. The domain I trusted with the drivers and the toolstack can ask me for the privileged things, and I will obey, because obedience to dom0 is the definition I was given. My isolation is perfect and beside the point if the trusted domain turns.

I can prove the guests are separated. I cannot prove the one I trust deserved it.

Ops & failure

  • xl / libxl: control; xenstore: live state.
  • xl dmesg + dom0 logs + xentrace: the evidence trail.
  • Noisy neighbor → contention, steal time spikes.
  • IRQ storms, softirq pressure, disabled interrupts.
  • Backend stall = unconsumed rings, timeouts.
  • dom0 compromise defeats guest isolation entirely.

Operations, observability, and failure modes

Operators manage Xen through the xl command and the libxl library, inspect runtime state via xenstore, and read hypervisor messages with xl dmesg alongside dom0 kernel and toolstack logs. Lower-level investigation uses xentrace and debug keys for scheduler, interrupt, and memory state.

Common failure modes include noisy-neighbor contention (a busy guest starving others, visible as elevated steal time), interrupt problems such as IRQ storms or unhandled IRQs that Xen disables, and softirq pressure in dom0 or driver domains. Split-driver failures appear as stalled backends, unconsumed ring entries, and frontends timing out.

The most serious risk is dom0 compromise: because dom0 holds privileged hypercalls, drivers, and the toolstack, its compromise undermines the isolation Xen provides between guests. Reducing dom0's trusted footprint—via driver domains, stub domains, and disaggregation—limits but does not eliminate this concentration of risk. Effective incident response correlates hypervisor traces, domain state, and host logs to localize whether the fault is in a guest, a backend, the scheduler, or the hardware.

Between your commands there are long stretches where nothing is asked of me. The guests block on I/O. The run queues drain. A core goes idle and I halt it to save power, and for a few hundred microseconds I am only waiting.
