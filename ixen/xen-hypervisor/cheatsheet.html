Domain Anatomy

Xen is a type-1 hypervisor that owns the CPU, MMU, and interrupts; everything else runs in a domain. The thin hypervisor delegates policy and most device handling to privileged guests rather than doing it itself.

dom0
First, privileged domain; runs toolstack, real device drivers, and hosts the default backends.
domU
Unprivileged guest; sees only virtual CPUs, memory, and paravirtual or emulated devices.
driver domain
Deprivileged domU that owns real hardware and exports backends, shrinking dom0's trust surface.
stub domain
Per-guest mini-domain running QEMU device emulation, isolating the emulator from dom0.
Note "Disaggregation" splits dom0 into driver and stub domains so a compromised backend cannot pivot to host-wide control.

Execution Modes

ModeHow it runsTrade-off
PVRing-deprivileged guest using hypercalls; no hardware virt needed.Fast historically; aging, large attack surface, removed for x86-64 in many builds.
HVMFull hardware virtualization (VT-x/AMD-V) with QEMU emulation.Runs unmodified OSes; emulation cost on boot/legacy I/O.
PVHHVM container with PV drivers, no emulated platform/BIOS.Modern default: lean, no QEMU, needs PVH-aware guest kernel.

HVM guests use PV-on-HVM drivers to bypass slow emulation for disk and net once booted, converging toward PVH behavior at runtime.

CPU Scheduling

xl sched-credit2
Default scheduler; weight-based proportional fair share across vCPUs.
xl sched-credit -d <dom> -w 512
Set a domain's CPU weight (relative share).
xl vcpu-pin <dom> 0 2
Pin vCPU 0 to physical CPU 2 for cache locality or isolation.
xl cpupool-list
Partition pCPUs into pools, each with its own scheduler.

The hypervisor multiplexes vCPUs onto pCPUs; cpupools hard-partition the machine, while weights and caps shape contention inside a pool.

Memory Translation & Sizing

Address layers

Guests use a per-domain P2M map; HVM/PVH rely on hardware nested paging (EPT/NPT), while PV used directly validated page tables. The hypervisor reserves a slice of every guest's address space.

Ballooning

The balloon driver returns or reclaims guest pages on demand, letting hosts oversubscribe RAM between memory and maxmem bounds.

xl mem-set <dom> 2048
Balloon a domain's current memory target to 2 GiB.
xl mem-max <dom> 4096
Set the hard ceiling the balloon can grow back to.
Warning Ballooning below the guest's working set triggers in-guest OOM; never set mem-set under what the workload actually touches.

Split I/O Plumbing

Paravirtual I/O is a frontend (in the guest) talking to a backend (dom0 or driver domain) over shared memory. Three primitives wire it together.

blkfront ↔ blkback
Disk; netfront ↔ netback handles networking via the same model.
grant references
Guest authorizes a backend to map/copy specific pages — the basis for zero-copy ring buffers.
event channels
Lightweight virtual interrupts that signal "ring has work" between domains.
xenstore
Shared hierarchical key-value DB used for device discovery, state handshakes, and config — not bulk data.
xenstore-ls /local/domain/3/device
xenstore-read /local/domain/3/device/vbd/51712/state
xenstore-write /local/domain/3/control/shutdown poweroff
Tip Grants + event channels are the transport; xenstore is the control plane. A stuck device often shows a frontend/backend state mismatch in xenstore.

Passthrough & IOMMU Isolation

xl pci-assignable-add 00:1f.2
Reserve a PCI device (detach from dom0) for passthrough.
xl pci-attach <dom> 00:1f.2
Hotplug a passthrough device into a running guest.
pci=['01:00.0']
Config-file form for boot-time assignment.
iommu=required
Xen cmdline; enforce VT-d/AMD-Vi so DMA is contained.

The IOMMU translates and confines device DMA so a passed-through NIC or GPU cannot read arbitrary host memory. Devices behind the same IOMMU group must be assigned together.
